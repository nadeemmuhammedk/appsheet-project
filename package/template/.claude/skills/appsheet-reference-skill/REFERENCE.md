## How to Use This Skill

### When to Invoke

**User-initiated invocation (explicit questions):**
- "How do I filter by current user?"
- "Show me SELECT formula syntax"
- "What parameters does a deck view have?"
- "How to create an action that updates status?"
- "What's the syntax for VALID_IF?"

**Automatic invocation (context-aware - NO USER REQUEST NEEDED):**
- User is building any AppSheet feature (tables, views, actions, slices)
- User is writing formulas (SELECT, VALID_IF, SHOW_IF, virtual columns)
- User is configuring views (need proper view settings)
- User is creating actions (need proper behavior and formulas)
- User is designing security (need filtering formulas)
- User is documenting AppSheet components (need accurate syntax)
- User mentions: "create view", "add formula", "configure action", "filter data"
- User describes AppSheet functionality even without explicitly asking for help

**IMPORTANT:** When building AppSheet features or documentation, you MUST automatically reference this documentation without waiting for user prompts. This ensures accurate configurations with exact parameter names and patterns.

### Response Pattern

When responding to a query:

1. **Identify the relevant component** from the user's question
2. **Read the corresponding documentation file** using the Read tool
3. **Extract relevant sections:**
   - Purpose (what the component does)
   - Syntax (formula or configuration syntax)
   - Parameters (available options)
   - Examples (code examples and usage patterns)
4. **Provide concise, actionable guidance:**
   - Reference specific syntax and parameters
   - Include code snippets where applicable
   - Highlight common gotchas or warnings
5. **Suggest related information** if the question spans multiple components

## Component-Specific Guidance

### Formula Best Practices

**When reading formula documentation, focus on:**
- Exact syntax (case-sensitive, parentheses, commas)
- Parameter order and types
- Return values
- Common patterns and gotchas

**Common gotchas:**
- SELECT filters: Boolean expression required in second parameter
- VALID_IF: Must return list of valid values
- SHOW_IF: Must return boolean (TRUE/FALSE)
- Column references: Use [ColumnName] syntax
- String comparisons: Case-sensitive by default

### View Best Practices

**When reading view documentation, focus on:**
- View type capabilities and limitations
- Required vs optional settings
- Display configuration options
- Security and SHOW_IF patterns
- Sorting, grouping, filtering options

**Common gotchas:**
- Calendar view: Requires date column
- Map view: Requires address or LatLong columns
- Chart view: Requires grouping and aggregation
- Dashboard view: Combines multiple existing views
- Form view: Shows input fields, not read-only

### Action Best Practices

**When reading action documentation, focus on:**
- Action behavior types (data vs navigation)
- Referenced Rows formula syntax
- Column value assignment syntax
- SHOW IF conditions
- Action prominence and icons

**Common gotchas:**
- Referenced Rows: Must return list of row keys
- Column values: Use [ColumnName] format
- SHOW IF: Boolean expression for visibility
- Set values action: Can update existing rows
- Add new row: Creates new record

### Security Filter Best Practices

**When reading security documentation, focus on:**
- USEREMAIL() function usage
- Row-level filtering formulas
- Table-level security settings
- Slice-based security

**Common gotchas:**
- Security filters: Applied at table level
- USEREMAIL(): Returns current user's email
- Row filtering: Boolean expression (TRUE = show row)
- Combine with slices: For view-specific security
